# Risk Register (Daftar Risiko)

Dokumen ini berisi daftar potensi risiko sistem finansial, diklasifikasikan berdasarkan dampak (Critical, High, Medium, Low).

| No | Modul | Temuan / Risiko | Tingkat Risiko | Dampak | Rekomendasi Solusi |
|---|---|---|---|---|---|
| 1 | Finance/Core | **Transaksi Bisa Di-update Secara Langsung (Mutable)**. Transaksi kas manual bisa diedit (nominal/tipe/kategori) via `update()`. | **Critical** | Memungkinkan manipulasi nominal tanpa persetujuan approver, walau dicatat di activity log. Sulit direkonsiliasi. | Larang `UPDATE` untuk field krusial (Nominal, Tipe) setelah transaksi *posted*. Gunakan mekanisme *Reversal* (Void lalu buat baru) atau Maker-Checker untuk proses edit. |
| 2 | Database | **Tipe Data Amount Menggunakan unsignedBigInteger**. | **Medium** | Aman dari error presisi *floating point*, namun tidak mendukung pencatatan nilai desimal jika suatu saat diperlukan (misal: pajak, interest, multi-currency). | Ubah tipe data di database menjadi `DECIMAL(19, 4)` atau tetap pertahankan Integer namun jadikan standar (menyimpan nilai mata uang x 100/1000). (Di Indonesia, integer sering digunakan karena IDR tidak ada sen. Jika ini tujuannya, cukup didokumentasikan). |
| 3 | Concurrency | **Race Condition pada Payment & Mutasi Kas**. | **High** | Memungkinkan "Double Submit" jika user mengklik tombol bayar berulang kali, mengakibatkan transaksi ganda sebelum transaksi pertama selesai divalidasi. | Implementasikan `Idempotency-Key` atau Lock database secara pesimistik saat `storePayment`. |
| 4 | Reporting | **Perhitungan Running Balance di Controller**. `TransactionsController@index` menghitung saldo berjalan di backend saat melooping paginasi. | **High** | Jika transaksi disortir berdasar field lain selain tanggal, atau difilter secara spesifik, `running_balance` berpotensi tidak akurat. | Gunakan sistem Ledger/Mutasi yang mem-persist saldo setiap transaksi, atau query agregat dari database (Window functions `SUM() OVER()`). |
| 5 | Authorization | **Gate/Policy belum menyeluruh**. Hardcode di `TransactionsController`: `!$request->user()->can('transactions.adjust.amount')`. | **Medium** | Tersebar di berbagai layer, bisa terlewat jika ada penambahan fitur. | Sentralisasi semua rule ke dalam `Policies`. |
| 6 | Database | **Ketiadaan Constraint Database untuk Periode**. Periode Iuran dicek melalui query Eloquent di Service, bukan DB Constraint (Unique Index per member & periode). | **Medium** | Jika terjadi *Race Condition*, bisa ada 2 pembayaran iuran pada bulan yang sama untuk user yang sama. | Tambahkan constraint `UNIQUE(member_id, period_ym)` di tabel `dues_payment_allocations`. |
